# 变更提案：数据库独立挂载与远端目录缓存增强

## 背景
当前服务默认将 SQLite 放在 `/app/data/app.db`，虽然可以通过挂载 `/app/data` 解决持久化，但数据库与其它运行时数据耦合在同一个目录中，不利于用户单独管理数据库文件。同时，远端目录文件存在性检查仍然主要依赖任务执行时即时拉取目录数据，缺少落本地缓存的持久层，导致容器重启后缓存无法复用。用户还提出希望在任务执行时可以控制是否强制同步远端目录文件，以便在性能与数据新鲜度之间做取舍。

## 目标
- 提供**数据库独立存储目录**，便于单独挂载和备份。
- 将**远端目录文件清单缓存到本地 SQLite**，避免每次任务都重复调用 115 接口。
- 在任务配置中新增**是否强制同步远端目录文件**开关，控制执行前是否刷新远端目录缓存。
- 保持现有“按目录 + 按文件名/按 SHA1 防重”的策略，同时改用本地缓存作为主要查询来源。

## 非目标
- 本次不引入 Redis、PostgreSQL 等外部缓存/数据库。
- 本次不实现多实例分布式缓存一致性。
- 本次不扩展为全量同步整个 115 网盘目录树，仅针对任务执行过程中访问到的目标目录做缓存。

## 用户价值
- 容器重建后数据库路径更清晰，运维更容易挂载与恢复。
- 远端目录缓存可复用，减少重复 API 调用，提升任务执行效率。
- 用户可根据实际需求选择“使用缓存优先”或“每次强制刷新远端目录”，兼顾性能与一致性。
