# 技术方案

## 方案概览
本次改造建议采用“**目录准备流程对齐插件，上传链路保留当前已对齐实现**”的方式，分离“目录发现/创建”和“文件上传”两个职责层。

推荐方案：**方案 B（流程对齐型重构，推荐）**

### 方案 A：最小改动兼容
- 思路：在现有 `precreate_remote_dirs()` / `prepare_dir_context()` 框架中插入插件式逐级建目录逻辑，仅替换 `_get_folder()` 的实现。
- 优点：改动面小，兼容当前批处理缓存结构。
- 缺点：调用顺序仍然保留“预创建叶子目录 + 目录级缓存同步”的框架，无法真正对齐插件目录上传方式。
- 结论：**不推荐**，只能做到“局部接口像插件”，做不到“整体流程像插件”。

### 方案 B：流程对齐型重构（推荐）
- 思路：
  - 新增插件式目录解析/创建服务，负责“查询父目录 → 从根逐级列目录 → 缺失则逐级创建”；
  - 主执行流程按“单文件视角”准备父目录；
  - 默认不再预创建叶子目录，不再强制先同步目录内全部文件；
  - 上传完成后轮询单文件存在性确认结果；
  - 现有 Open 上传实现保留，用作插件上传补丁语义的等价实现。
- 优点：最接近用户要求的“方式跟插件一样”，可解释性强。
- 缺点：会削弱当前批量任务场景下的目录级缓存收益，需要重新设计防重策略。
- 结论：**推荐**。

### 方案 C：双模式并存
- 思路：保留现有“批处理优化模式”，新增“plugin_aligned”执行模式，通过配置切换。
- 优点：风险更低，可灰度验证。
- 缺点：系统复杂度上升，后续维护两套执行语义。
- 结论：如你希望先稳妥迁移，可作为落地阶段的折中方案。

## 设计决策

### ADR-2026-04-03-01：目录准备流程从“路径 ensure”改为“逐级探测 + 逐级建目录”
- **状态**：建议采纳
- **背景**：当前 `ensure_remote_dir()` 使用 `fs_dir_getid + fs_makedirs_app` 一次性确保整条路径存在，与插件目录上传主流程不一致。
- **决策**：新增插件对齐的目录准备器，执行顺序调整为：
  1. 先尝试查询目标父目录；
  2. 若不存在，则从根目录开始；
  3. 每一级目录先列举当前目录子项；
  4. 有同名目录则复用；
  5. 无同名目录则创建当前级目录；
  6. 迭代直到目标父目录可用。
- **影响**：`run_service`、`upload_strategy`、`p115 gateway` 的职责边界需要调整。

### ADR-2026-04-03-02：默认流程取消“预创建叶子目录 + 目录级全量同步”
- **状态**：建议采纳
- **背景**：插件目录上传主流程不是先按目录做批处理预热，而是按目标文件路径即时准备目录并直接上传。
- **决策**：
  - 默认主路径移除 `precreate_remote_dirs()`；
  - `prepare_dir_context()` 不再作为默认前置动作；
  - 远端目录文件全量同步改为可选能力，仅在需要执行目录级防重/巡检时启用。
- **影响**：远端缓存命中逻辑需要从“目录级前置同步”调整为“按需调用”。

### ADR-2026-04-03-03：上传结果确认改为“上传后轮询单文件存在性”
- **状态**：建议采纳
- **背景**：插件目录上传主流程上传完成后会轮询目标文件路径确认最终结果，而当前项目更多依赖上传响应与目录缓存回写。
- **决策**：新增上传后确认器：
  - 按目标远端文件路径轮询 1~3 次；
  - 成功则补齐 `remote_file_id/pickcode` 并落库；
  - 失败则保留上传响应并标记为“上传已提交但确认失败/待确认”。
- **影响**：日志阶段、结果消息、失败语义需要更细化。

## 模块改造设计

### 1. `backend/app/services/run_service.py`
**现状问题：**
- 先对全部候选文件分组；
- 调用 `precreate_remote_dirs()`；
- 再逐目录 `prepare_dir_context()`；
- 流程明显偏批处理，不像插件目录上传。

**改造方向：**
- 将主循环改为“逐文件执行”或“轻量分组但逐文件准备目录”。
- 新顺序建议：
  1. 扫描候选文件；
  2. 对每个文件计算目标远端路径；
  3. 调用目录准备器确保父目录存在；
  4. 执行上传；
  5. 轮询确认远端文件；
  6. 再做结果落库。
- 如保留分组，仅用于日志展示与限流，不再用于提前预创建目录。

### 2. `backend/app/services/upload_strategy.py`
**现状问题：**
- `_get_folder()`、`precreate_remote_dirs()`、`prepare_dir_context()` 都围绕“目录级缓存/预创建”设计；
- 上传前防重过度依赖 `context.items`。

**改造方向：**
- 保留 `upload_candidate_in_context()` 中“秒传 → 分片/Open 上传”的核心逻辑；
- 拆出以下新对象：
  - `PluginAlignedFolderResolver`：负责逐级列目录 / 建目录；
  - `UploadedFileVerifier`：负责上传后按路径轮询确认；
  - `RemoteDuplicateChecker`：负责按需查询单文件/单目录，而不是默认预热整目录。
- `_get_folder()` 可保留为兼容方法，但主路径不再依赖它。

### 3. `backend/app/integrations/p115/client.py`
**现状问题：**
- 缺少“列举某父目录下子目录并按名称查找”的清晰接口；
- 缺少“创建当前级目录（类似 `fs_mkdir`）”和“按路径轮询单文件”的业务友好封装。

**改造方向：**
新增或重构以下网关方法：
- `get_dir_id_by_path(path)`：包装 `fs_dir_getid/fs_dir_getid_app`。
- `list_dir_entries(pid, include_dirs=True)`：支持列目录及目录项筛选。
- `find_child_dir(parent_id, name)`：在某目录下查找同名子目录。
- `create_child_dir(parent_id, name)`：优先对齐插件的 `fs_mkdir` 语义。
- `ensure_remote_dir_plugin_style(path)`：执行逐级探测与逐级创建。
- `get_remote_item_by_path(path)` 或 `poll_remote_file(path)`：上传后确认结果。

### 4. `backend/app/services/remote_dir_cache_service.py`
**改造方向：**
- 重新定义缓存用途：从“上传前防重预热”转为“结果回写与可选加速”。
- 若采用方案 B，默认流程中缓存不再是核心依赖，只作为可选优化层。

### 5. 配置与模型
建议新增一个执行策略开关，便于灰度：
- `upload_flow_mode = plugin_aligned | batch_cached`

默认可先设为：
- 开发阶段：`plugin_aligned`
- 如需平稳迁移：先默认 `batch_cached`，待验证后切换

对应影响：
- `app/models/source.py`
- `app/schemas/source.py`
- `app/repositories/source_repository.py`
- 前端同步源编辑页（如需配置暴露）

## API 对齐策略

### 目录准备 API
推荐按插件风格调整为：
1. `fs_dir_getid` / `fs_dir_getid_app`：判断目标目录或父目录是否存在
2. `fs_files`：列出当前父目录的子项（含目录）
3. `fs_mkdir`：创建当前缺失级目录

### 上传 API
保留当前已对齐能力：
- 秒传：`upload_file_init`
- Open：`/open/upload/init` → 二次校验 → `/open/upload/get_token` → `/open/upload/resume` → OSS 分片上传

### 上传后确认 API
新增或显式使用：
- `get_item by path` 风格能力
- 或基于目录列举 + 文件名匹配实现单文件轮询确认

## 安全与性能

### 风险
1. **性能回退风险**：从目录批处理切到逐文件准备目录后，大批量同步请求次数可能上升。
2. **兼容性风险**：现有依赖目录缓存的防重逻辑可能失效或行为变化。
3. **状态一致性风险**：上传成功但轮询确认失败时，状态语义需要明确。

### 规避措施
1. 通过 `upload_flow_mode` 开关灰度发布，保留旧模式回退能力。
2. 将目录准备、上传、确认三个阶段拆开打日志，便于对比插件行为。
3. 为目录对齐流程补充集成测试：
   - 目录完全存在；
   - 多级目录部分不存在；
   - 上传成功但确认延迟；
   - 取消发生在目录创建/分片上传/确认阶段。
4. 在知识库中明确说明“plugin_aligned 模式牺牲部分批量效率，换取流程一致性”。

## 验证计划
1. 单元测试：
   - 逐级目录解析与创建顺序；
   - 缺失目录创建时 API 调用序列；
   - 上传后轮询确认逻辑。
2. 集成测试：
   - 本地嵌套目录上传到空远端路径；
   - 同路径重复上传；
   - 秒传命中与未命中两类场景。
3. 对照验证：
   - 将一组样例文件在 `115helper` 与插件中分别执行；
   - 对比日志顺序、建目录行为与最终文件落点。
