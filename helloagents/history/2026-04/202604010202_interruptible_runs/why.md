# 变更提案: 可中断后台任务执行

## 需求背景
当前系统已经完成后台异步任务执行，但“取消任务”仍属于软取消：用户点击取消后只会修改运行状态，后台线程未真正停止文件扫描或上传处理，导致任务可能继续执行一段时间，日志与状态也会出现偏差。用户要求将取消功能升级为真正可中断的后台任务，使后台执行线程收到取消请求后，在可控检查点停止后续处理。

## 产品分析

### 目标用户与场景
- **用户群体:** 需要在任务执行中及时终止错误任务或误触发任务的本地自部署用户。
- **使用场景:** 用户发现同步目录选错、上传策略设置有误或任务耗时过长，希望立即停止该任务。
- **核心痛点:** 当前取消按钮不能真正阻止后台继续处理，用户会看到状态已取消但日志和文件处理仍在继续。

### 价值主张与成功指标
- **价值主张:** 让取消操作变成真正可靠的执行控制能力，而不是单纯的状态标记。
- **成功指标:**
  - 用户点击取消后，后台任务尽快停止。
  - 已完成处理的文件记录保留，未处理文件不再继续。
  - 运行状态最终为 `cancelled`，且实时日志明确显示中断过程。
  - 被取消任务会释放任务占用，允许用户后续重新发起任务。

### 人文关怀
取消是用户恢复掌控感的重要操作。系统应明确告诉用户“取消请求已收到”“任务已在后台停止”，而不是让用户处于不确定状态。

## 变更内容
1. 为后台执行器增加取消信号注册与查询机制。
2. 在运行服务中增加中断检查点，确保扫描后、逐文件处理前后都能响应取消。
3. 调整取消接口，使其发送取消信号并统一更新日志/SSE 状态。
4. 前端沿用现有实时日志与状态页，无需大幅重构。

## 影响范围
- **模块:** backend-api、job-scheduler、sync-engine、frontend-console
- **文件:** 预计修改 `backend/app/services/async_run_executor.py`、`backend/app/services/run_service.py`、`backend/app/api/runs.py`，并补充测试与文档
- **API:** 取消接口语义增强，但路径保持不变
- **数据:** 无需新增表

## 核心场景

### 需求: R1 Cancel Signal
**模块:** backend-api, async-executor
用户点击取消后，系统需要登记取消信号。

#### 场景: Cancel Pending Run
任务还在 `pending` 状态时。
- 任务不再启动执行。
- 状态直接变为 `cancelled`。

#### 场景: Cancel Running Run
任务已进入 `running` 状态时。
- 后台线程在下一检查点检测到取消信号。
- 停止后续处理并退出执行循环。

### 需求: R2 Interrupt Checkpoints
**模块:** run-service, sync-engine
后台执行必须在关键步骤响应取消。

#### 场景: Stop Before Next File
用户在处理多个文件时点击取消。
- 当前文件处理完成后，后续文件不再开始。
- 日志记录中断发生点。

### 需求: R3 Consistent Visibility
**模块:** frontend-console, backend-api
取消后的状态、日志与 SSE 必须一致。

#### 场景: Cancelled With Logs
用户取消任务后。
- 运行详情页看到“取消请求已收到”“任务执行已中断”等日志。
- SSE 推送最终状态 `cancelled`。

## 风险评估
- **风险:** 正在进行的单个上传操作若不可中断，可能只能在当前文件完成后停止。
- **缓解:** 首版明确采用“检查点中断”策略，保证不会启动后续文件；后续如需更强能力，再研究上传过程内的细粒度中断。
- **风险:** 取消信号清理不及时可能导致后续同一 `run_id` 状态混乱。
- **缓解:** 执行器在任务结束、取消完成后统一清理运行上下文。
