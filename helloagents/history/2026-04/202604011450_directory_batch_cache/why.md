# 变更提案：目录级批处理远端缓存与防重性能优化

## 背景
当前系统虽然已经将 115 远端目录文件缓存持久化到 SQLite，并支持按文件名/按 SHA1 防重，但执行流程仍然以“逐文件”为主。实际表现为：

- 同一远端目录下的多个文件，仍然逐个进入上传策略逻辑
- 即使缓存已存在，日志上仍能观察到明显的单文件间隔
- 在“按文件名跳过”模式下，系统仍会提前计算本地文件 SHA1，导致大文件场景下耗时明显
- 目录确认与目录缓存装载虽然有一定复用，但没有在任务执行层面做显式的“按目录分组批处理”

用户希望进一步优化为：
- 本地 SQLite 中按目录组织远端文件缓存
- 同目录文件一批处理
- 同目录避免重复调用远端接口获取目录
- “按文件名跳过”模式下尽量避免不必要的 SHA1 计算

## 目标
- 将任务执行主流程升级为**按目标远端目录分组批处理**。
- 同目录文件共享一次目录解析、一次缓存加载、一次可选远端刷新。
- 在 `duplicate_check_mode=name` 时，命中前不计算 SHA1。
- 明确区分“目录批次日志”和“文件处理日志”，提升可观察性。

## 非目标
- 本次不调整数据库表结构的核心模型，只优化执行路径与缓存使用方式。
- 本次不实现目录缓存 TTL 自动过期机制。
- 本次不引入队列化并发目录处理。

## 预期收益
- 同目录多文件场景下显著减少重复逻辑开销。
- 在“按文件名跳过”场景下降低大文件扫描成本。
- 日志中能够清晰看到目录级批处理行为与缓存命中情况。
