# 变更提案: 多任务同步与任务级日志增强

## 需求背景
当前项目已经具备单机部署的 115 同步控制台基础骨架，但在产品语义上仍偏向“基础同步源配置 + 运行记录”。用户进一步明确要求：系统必须支持多个同步任务，每个任务对应不同本地源目录与不同 115 目标目录，并可分别配置独立的定时规则；同时需要一个统一的任务管理视图，能够查看和管理这些定时任务，并按任务级别记录执行历史、开始/结束时间、执行状态以及执行过程中的日志。

## 产品分析

### 目标用户与场景
- **用户群体:** 同时维护多套媒体目录、资料目录或备份目录的 115 自部署用户。
- **使用场景:** 用户希望把电影、剧集、纪录片、资料等目录拆分成多个独立同步任务，分别同步到 115 中不同目标目录，并分别设定执行频率。
- **核心痛点:** 现有系统虽有 `SyncSource` 概念，但前后端在“任务管理”“独立调度状态”“任务级日志查看”方面表达不完整，无法满足多任务场景下的运维观察需求。

### 价值主张与成功指标
- **价值主张:** 把“同步源”正式提升为“可管理的同步任务”，让每个任务拥有独立配置、独立调度、独立运行历史与独立日志视图。
- **成功指标:**
  - 能同时维护多个同步任务。
  - 每个任务可独立配置本地目录、115 目录、上传模式、过滤规则与 Cron。
  - 前端可查看每个任务的启用状态、最近执行时间、下次执行信息、最近运行结果。
  - 每次执行都能记录任务级日志，并在任务详情/运行详情中展示。

### 人文关怀
对于多任务场景，系统应尽量降低排障门槛：日志信息应可读、分层、可追溯，避免用户只能看到“失败”而不知道失败发生在哪个阶段；同时仍需保持 Cookie 等敏感信息不进入日志与前端展示。

## 变更内容
1. 明确“同步任务”模型与 UI 语义，支持多个独立任务管理。
2. 增强调度管理能力，展示任务启停状态、Cron、最近执行时间与下次执行计划。
3. 新增任务级日志模型与日志写入机制。
4. 扩展运行详情，支持按任务级查看执行过程日志。
5. 调整前后端页面结构，使任务管理与运行历史管理更清晰。

## 影响范围
- **模块:** backend-api、frontend-console、sync-engine、job-scheduler、p115-gateway
- **文件:** 预计修改 `backend/app/models/*`、`backend/app/services/*`、`backend/app/api/*`、`frontend/src/views/*`、`frontend/src/types/*`、`frontend/src/api/*`
- **API:** 扩展同步任务管理、调度信息查询、日志查询接口
- **数据:** 预计新增任务日志表，并扩展任务/运行记录字段

## 核心场景

### 需求: R1 Multi Task Management
**模块:** backend-api, frontend-console
系统需要支持多个同步任务，并将每个任务作为独立实体进行管理。

#### 场景: Multiple Independent Tasks
用户创建多个同步任务。
- 每个任务拥有不同本地目录与不同 115 目标目录。
- 每个任务的配置互不影响。

#### 场景: Task Overview
用户在任务列表页查看所有任务。
- 能看到任务名称、源目录、目标目录、启用状态、Cron、最近执行结果。
- 能对单个任务执行编辑、启停、手动运行操作。

### 需求: R2 Independent Scheduling
**模块:** job-scheduler, backend-api, frontend-console
每个任务都需要可配置独立的定时规则，并在界面中进行管理。

#### 场景: Per Task Cron
用户为不同任务设置不同 Cron。
- 调度器按任务维度注册和更新调度。
- 修改任务配置后对应调度即时刷新。

#### 场景: Scheduler Visibility
用户查看任务调度状态。
- 能看到每个任务的 Cron、启用状态、最近执行时间、下次计划时间。
- 无效或禁用任务在界面上有清晰状态标识。

### 需求: R3 Task Level Execution Record
**模块:** backend-api, sync-engine, job-scheduler
系统需要记录每个任务的执行历史和关键时间信息。

#### 场景: Run History By Task
用户查看某个任务的运行记录。
- 能看到每次执行的开始时间、结束时间、触发方式、状态和统计摘要。
- 运行记录按任务维度过滤和展示。

#### 场景: Latest Run Summary
用户在任务列表页查看最近执行结果。
- 能快速知道最近一次运行是否成功、部分失败或失败。
- 无需进入详情也能判断任务健康状况。

### 需求: R4 Task Level Log Display
**模块:** sync-engine, backend-api, frontend-console
系统需要在任务执行过程中记录并展示任务级日志。

#### 场景: Structured Execution Log
任务执行过程中写入结构化日志。
- 至少记录阶段：任务开始、扫描目录、过滤结果、目录创建、秒传尝试、分片上传、完成、失败。
- 每条日志应带时间、任务 ID、运行 ID、日志级别和消息。

#### 场景: View Logs In UI
用户在前端查看任务运行日志。
- 可在运行详情中按任务维度查看日志列表。
- 日志按时间排序，便于定位问题。

### 需求: R5 Safe Logging
**模块:** backend-api, p115-gateway
日志与展示需要兼顾可用性和安全性。

#### 场景: Sensitive Info Protection
系统记录任务日志时。
- 不记录 Cookie、授权头、敏感请求体等明文信息。
- 错误日志优先记录业务可读信息而非裸异常堆栈。

## 风险评估
- **风险:** 当前运行模型是同步执行，日志虽可落库，但难以实现真正实时体验。
- **缓解:** 本阶段优先实现“任务级持久日志 + 运行后查询”，必要时预留未来 WebSocket/SSE 扩展点。
- **风险:** 调度信息（如下次执行时间）依赖 APScheduler 运行状态，应用重启或任务配置错误时可能不准确。
- **缓解:** 统一由调度服务生成状态快照，并在接口层返回标准化字段。
