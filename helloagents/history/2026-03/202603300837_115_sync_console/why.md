# 变更提案: 115 同步控制台初始化

## 需求背景
当前仓库仅有占位 README，尚未形成可运行的同步系统。用户希望参考 `p115client` 与 `/mnt/MoviePilot-Plugins/` 中的现有实践，在本地建立一个前后端项目，通过 SQLite 保存配置与运行记录，实现本地目录到 115 网盘的同步，并支持手动/定时执行、秒传/分片上传策略以及文件后缀过滤。

## 产品分析

### 目标用户与场景
- **用户群体:** 本地自部署的 115 网盘重度用户、媒体资源整理用户。
- **使用场景:** 需要把指定目录中的新资源持续同步到 115，并能从网页查看执行情况。
- **核心痛点:** 手工上传效率低、秒传与分片上传策略切换麻烦、缺少可视化状态与重试能力。

### 价值主张与成功指标
- **价值主张:** 用单机可部署的前后端控制台，把 115 目录同步变成可配置、可审计、可重试的持续任务。
- **成功指标:** 能成功创建同步源、手动触发运行、按 Cron 自动执行、正确应用上传模式与后缀过滤、在前端看到运行结果。

### 人文关怀
系统默认面向单用户本地部署，优先保护 Cookie 等敏感信息，不在界面和日志中暴露明文凭证，降低误操作与信息泄露风险。

## 变更内容
1. 初始化前后端项目骨架与本地开发运行方式。
2. 建立 SQLite 数据模型，持久化同步源、运行记录和文件结果。
3. 封装 115 网盘同步引擎，支持秒传/分片上传策略与后缀过滤。
4. 提供手动执行与 Cron 定时执行能力。
5. 提供基础前端控制台，用于配置、查看与触发任务。

## 影响范围
- **模块:** backend-api、frontend-console、sync-engine、job-scheduler、p115-gateway
- **文件:** 预计新增 `backend/*`、`frontend/*`、`docker-compose.yml`、`.env.example`、`scripts/*`
- **API:** 新增同步源管理、任务运行、系统设置接口
- **数据:** 新增 SQLite 表 `sync_sources`、`job_runs`、`file_records`、`app_settings`

## 核心场景

### 需求: R1 Source Config
**模块:** backend-api, frontend-console
用户需要在控制台中维护同步源，包括本地目录、115 目标目录、上传模式、后缀规则与定时表达式。

#### 场景: Create Source
用户填写配置并保存。
- 配置通过校验并写入 SQLite。
- 若配置启用了 Cron，则系统重建对应调度任务。

#### 场景: Edit Source
用户修改现有配置。
- 系统更新数据库记录。
- 调度器按最新配置重新注册任务。

### 需求: R2 Sync Execution
**模块:** sync-engine, p115-gateway, job-scheduler
系统需要能手动或定时触发同步，把本地文件按相对路径映射上传到 115。

#### 场景: Manual Run
用户在前端点击“立即同步”。
- 系统创建运行记录并启动后台任务。
- 同步过程状态可被前端查询。

#### 场景: Cron Run
到达 Cron 时间点时自动执行同步。
- 系统按配置触发运行。
- 同一同步源不会出现并发重复执行。

### 需求: R3 Upload Strategy
**模块:** sync-engine, p115-gateway
系统需要支持秒传与分片上传两类策略。

#### 场景: Fast Only
配置为仅秒传。
- 系统执行 `upload_file_init` 复用校验。
- 未命中秒传时记录为跳过或失败，不进入真实上传。

#### 场景: Fast Then Multipart
配置为秒传优先，失败后回退分片上传。
- 系统先尝试秒传。
- 秒传未复用时自动继续分片上传。

#### 场景: Multipart Only
配置为仅分片上传。
- 系统跳过秒传判定分支中的复用成功判定。
- 直接通过分片上传完成文件传输。

### 需求: R4 Suffix Filter
**模块:** sync-engine, frontend-console
系统需要按文件后缀过滤候选文件。

#### 场景: Include Suffixes
用户配置白名单后缀。
- 仅匹配后缀的文件进入同步流程。
- 过滤掉的文件在运行详情中可追踪统计。

#### 场景: Mixed Case Suffix
文件扩展名存在大小写差异。
- 系统按不区分大小写规则进行匹配。
- 同一后缀规则在 UI 与后端保持一致。

### 需求: R5 Run Visibility
**模块:** backend-api, frontend-console
用户需要在前端查看任务执行结果并重试失败项。

#### 场景: View Run Detail
用户打开某次运行详情。
- 能看到总文件数、秒传成功数、分片成功数、跳过数、失败数。
- 能按结果类型筛选文件明细。

#### 场景: Retry Failed
用户重试失败项。
- 系统仅重新调度失败文件或整次任务。
- 新运行记录与历史保持可追溯。

## 风险评估
- **风险:** 115 接口行为与客户端版本可能波动，导致上传策略实现不稳定。
- **缓解:** 首版优先沿用 `/mnt/MoviePilot-Plugins/` 已验证调用方式，封装统一适配层，并固定依赖版本。
- **风险:** 大文件分片上传中断或任务取消可能造成状态不一致。
- **缓解:** 记录运行状态与文件级结果，设计可恢复/可重试的任务模型。
